## 1. 地図と現場
**地図**は、あなたがClaudeに渡すもの＝プロンプト・スキル・コンテキスト。**現場**は、実際に仕事が起きる場所＝コードベースや現実の制約。この2つの**差分が「未知(unknowns)」**になる。✅

任せる仕事が大きいほど未知は増え、AIは「たぶんこうだろう」と推測で埋める。ここでズレる現象が**目標ドリフト**。✅ ズレはゼロにできない。だからこそ“差分を減らす工夫”と“フィードバック”が鍵になる。🔶

## 2. なぜ今Fableなのか
記事の中核的な主張は「Fableは、仕事の質が“自分の未知を明確にする力”でボトルネックになる最初のモデルだ」というもの。🔶 モデルの実行力が十分に高くなった結果、詰まる場所が**人間側に移った**という捉え方。

- これまで：モデルの能力が足りず詰まる → 「もっと賢いモデル」を待つ
- Fable時代：モデルは差分を上手く埋める → 「自分の未知を出せるか」で差がつく
- ⚠️ モデルの優劣は体感・主張ベース。学びは「人間側の準備が効く」点に置く。

## 3. 未知の4象限
問題をClaudeに持ち込む前に、自分の知識を4つに棚卸しする。危険なのは下段（無自覚）。✅

| 象限 | 意味 | 扱い |
|---|---|---|
| 既知の既知 | プロンプトに書けること | まず充実させる |
| 既知の未知 | 未決だと自覚している問い | 具体化する |
| 無自覚の既知 | 自明すぎて書かないが見れば分かる前提 | 言語化して共有 |
| 無自覚の未知 | 考えたことすらないこと（最危険） | 探して照らす |

意識の向け方は ①→②→③→④。下段ほど見えないままだと危険。

## 4. 未知の絞り方と中庸バランス
言葉を細かくするほどAIが取り得る範囲が狭まり、未知が小さくなる（人間→男→頭も良くてかっこいい男…と絞れる例）。🔶

ただし両端は逆効果。**具体的すぎ**＝遊び（改善余地）が消え機械的に追従／**曖昧すぎ**＝一般的な“最善”で埋まりAIスロップになりがち。狙いは中庸。さらに“解像度の高いアイデア”に価値がある — 世間の一般ラインより一歩深く点を打つこと。

絞り込みは“制約を積み上げる”ことではない。「production品質で」と丸投げせず、**寿命・目的・利用者・失敗時の影響**といった**文脈**を渡すと、AIは自分で妥当な判断を選べる。🔶

## 5. 未知を潰す8つの技法
前・中・後の反復ループで使う。✅

- 実装前：①盲点チェック ②ブレスト＆プロトタイプ ③インタビュー ④リファレンス ⑤実装プラン
- 実装中：⑥実装ノート
- 実装後：⑦ピッチ（説明資料） ⑧マージ前クイズ

※ 元動画では「5つ」と表現。図解・原文の並びに合わせ、本資料は**8つ**で統一。⚠️（数え方の違いに注意）

## 6. 実例 動画編集
Fable紹介の映像を Claude Code で組み立てた例。著者は動画編集に詳しくなかったが、「未知を自覚に変えれば進める」を実演している。🔶

①既知から開始（できることは知っている・精度は未知）→ ②仕組みを教わる（カットの可否などを確認）→ ③試作で確かめる → ④色補正の良し悪しは“教わる側”に切替。手を動かすほど未知が発掘され、次の判断に戻ってくる。
